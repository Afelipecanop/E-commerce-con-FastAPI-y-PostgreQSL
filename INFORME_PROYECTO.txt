1. DESCRIPCIÓN GENERAL
================================================================================

E-commerce funcional desarrollado con FastAPI en el backend, PostgreSQL como motor
de base de datos y un frontend en HTML/CSS/JavaScript vanilla. El proyecto permite
registrar usuarios, administrar productos, usar un carrito de compras, procesar pagos
con Stripe y gestionar el contenido visual de la tienda mediante un panel de
administración.

Estado actual: aproximadamente 92% completado.
- Backend: ~95% funcional
- Frontend público: ~90% funcional
- Panel de administración: ~85% funcional
- Integraciones externas: Stripe, IA y despliegue configurados
- Testing automatizado: 0% (sin suite de pruebas implementada)

El proyecto se encuentra en una fase madura de MVP, con los módulos principales
operativos y listos para uso o para continuar con mejoras de UX, seguridad y
validaciones adicionales.

3. ARQUITECTURA TÉCNICA
================================================================================

3.1 BACKEND - FastAPI + PostgreSQL
------------------------------------

BASE DE DATOS:

TABLE: users
- id (UUID, PK)
- email (unique, indexed)
- hashed_password
- full_name
- is_admin (default: false)
- is_active (default: true)
- created_at
Relaciones: 1-1 con carts, 1-N con orders

TABLE: products
- id (UUID, PK)
- name
- description
- price
- stock
- image_url
- is_active (default: true)
- created_at
Relaciones: 1-N con cart_items y order_items

TABLE: carts
- id (UUID, PK)
- user_id (FK, unique)
- updated_at
Relaciones: 1-N con cart_items

TABLE: cart_items
- id (UUID, PK)
- cart_id (FK)
- product_id (FK)
- quantity
- added_at

TABLE: orders
- id (UUID, PK)
- user_id (FK)
- status
- total_amount
- stripe_payment_id
- created_at
Relaciones: 1-N con order_items

TABLE: order_items
- id (UUID, PK)
- order_id (FK)
- product_id (FK)
- quantity
- unit_price

TABLE: store_layout
- id (String, PK)
- block_type
- order_index
- config (JSON guardado como texto)
- is_visible
- created_at
- updated_at

Nota importante: en la implementación actual, la configuración visual se gestiona
mediante la tabla store_layout, guardando los datos de cada bloque como JSON dentro
de la columna config.

MODELOS Y LÓGICA DE NEGOCIO:

User:
- Manejo de credenciales con password hasheado
- Atributos de administración y activación de cuenta

Product:
- Catálogo de productos con precio, stock y estado activo/inactivo

Cart y CartItem:
- Carrito único por usuario
- Validación de stock antes de agregar o modificar items
- Cálculo del total del carrito

Order y OrderItem:
- Registro de órdenes con snapshot del precio unitario al momento de la compra
- Estados pending, paid y cancelled
- Integración con Stripe para confirmación final

StoreLayout:
- Bloques dinámicos para la tienda
- Soporte para reordenar, ocultar y editar secciones

RUTAS API:

ROUTER: /auth
- POST /register → crear usuario y carrito automático
- POST /login → devolver token JWT
- GET /me → datos del usuario autenticado

ROUTER: /products
- GET / → listar productos activos
- GET /{id} → detalle de producto
- POST / → crear producto (solo admin)
- PUT /{id} → actualizar producto (solo admin)
- DELETE /{id} → desactivar producto (solo admin)

ROUTER: /cart
- GET / → obtener carrito actual del usuario
- POST /items → agregar producto al carrito
- PUT /items/{id} → editar cantidad de un item
- DELETE /items/{id} → eliminar un item

ROUTER: /payments
- POST /checkout → crear sesión de pago en Stripe
- POST /webhook → confirmar pago y actualizar stock/orden
- GET /orders → historial de órdenes del usuario
- GET /orders/{id} → detalle de una orden

ROUTER: /layout
- GET / → devolver bloques actuales del layout
- PUT / → reemplazar el layout completo
- POST /blocks → agregar un bloque nuevo
- DELETE /blocks/{id} → eliminar un bloque
- POST /ai-generate → generar HTML/CSS con IA para contenido visual

SERVICIOS Y COMPONENTES PRINCIPALES:

- Auth service:
  - Hashing de contraseñas
  - Creación y validación de JWT
  - Autenticación por email/contraseña

- Middleware de seguridad:
  - get_current_user() protege endpoints de usuario autenticado
  - get_current_admin() protege operaciones administrativas

- Integración con Stripe:
  - Creación de checkout sessions
  - Manejo de webhooks
  - Actualización del estado de órdenes y stock

- Integración con IA:
  - Generación de bloques personalizados en HTML/CSS
  - Uso de Anthropic Claude vía API
  - Disponible solo para usuarios administradores

AUTENTICACIÓN Y AUTORIZACIÓN:

- JWT en header Authorization usando Bearer Token
- Protección de endpoints por roles y estado de sesión
- Control de acceso basado en el campo is_admin del usuario
- El registro genera un carrito vacío automáticamente

MIGRACIONES ALEMBIC:

- v1: tablas base del e-commerce (users, products, carts, cart_items, orders, order_items)
- v2: tabla store_layout para bloques visuales y configuración dinámica

4. FLUJOS PRINCIPALES
================================================================================

4.1 FLUJO DE COMPRA (CUSTOMER):

1. REGISTRO O LOGIN
   → El usuario crea una cuenta o inicia sesión
   → El backend valida las credenciales y devuelve un token JWT
   → El frontend almacena el token para las siguientes peticiones

2. EXPLORACIÓN DE PRODUCTOS
   → El cliente accede al catálogo público
   → El frontend obtiene los productos desde /products/
   → Se renderiza la vista con precio, stock y descripción

3. AGREGAR AL CARRITO
   → El usuario pulsa “Agregar al carrito”
   → El backend valida la existencia del producto y el stock disponible
   → Se crea o actualiza el item dentro del carrito del usuario

4. REVISAR CARRITO
   → El cliente accede a cart.html
   → Se consulta /cart/ y se muestran los items y el total
   → El usuario puede modificar cantidades o eliminar artículos

5. CHECKOUT
   → Al confirmar la compra, el frontend manda a /payments/checkout
   → El backend crea una orden en estado pending
   → Se genera una sesión de pago en Stripe y se devuelve la URL

6. PAGO Y CONFIRMACIÓN
   → El usuario completa el pago en Stripe
   → El webhook de Stripe confirma la transacción
   → El backend actualiza la orden a paid, disminuye stock y vacía el carrito


4.2 FLUJO ADMIN (MANAGEMENT):

1. LOGIN COMO ADMIN
   → Se autentica un usuario con permisos de administrador
   → El token JWT permite acceder a rutas protegidas

2. ACCESO AL PANEL
   → El usuario entra a admin.html
   → Se cargan las vistas de Editor, Productos y Ajustes
   → El panel permite crear, modificar y eliminar bloques visuales

3. GESTIÓN DE PRODUCTOS
   → Se pueden crear, editar y desactivar productos desde la vista administrativa
   → Los cambios se reflejan en el catálogo público

4. EDITOR DE LAYOUT
   → El administrador puede añadir bloques de tipo anuncio, hero, productos,
     texto, testimonios, categorías, imagen y footer
   → Puede reordenar secciones, editar propiedades y previsualizar el resultado
   → También puede generar secciones con IA a partir de un prompt

5. GUARDADO DE CAMBIOS
   → El layout se guarda mediante la API /layout/
   → Los bloques se almacenan en la base de datos y se vuelven visibles en la tienda
