1. DESCRIPCIÓN GENERAL
================================================================================

Proyecto e-commerce en desarrollo activo, con backend desarrollado en FastAPI,
PostgreSQL como motor de base de datos y un frontend en HTML, CSS y JavaScript
vanilla. La plataforma permite gestionar usuarios, catálogo de productos, categorías,
carrito, pagos, checkout invitado, contenido visual de la tienda, páginas de producto
personalizadas, ajustes de configuración y una experiencia de compra más completa
para clientes y administradores.

Estado actual: aproximadamente 98% del MVP funcional.
- Backend: ~97% funcional
- Frontend público: ~96% funcional
- Panel de administración: ~95% funcional
- Integraciones externas: activas y en uso
- Testing automatizado: 0% (sin suite implementada aún)

El proyecto se encuentra en una etapa madura de desarrollo, con módulos principales
operativos y con una base sólida para seguir escalando con nuevas mejoras y una
implementación más robusta en producción.

5. INFRAESTRUCTURA DEL PROYECTO
================================================================================

5.1 ARQUITECTURA DE INFRAESTRUCTURA
------------------------------------

El proyecto está compuesto por tres capas principales:
- Capa de frontend: archivos estáticos HTML/CSS/JS servidos desde un hosting estático.
- Capa de backend: API REST desarrollada con FastAPI y ejecutada mediante Uvicorn.
- Capa de datos: base de datos PostgreSQL gestionada por un servicio relacional.

La comunicación entre el frontend y el backend se realiza mediante peticiones HTTP
seguras, con autenticación por token JWT y configuración de CORS para permitir el
acceso desde entornos locales y de despliegue.

5.2 COMPONENTES DE DESPLIEGUE
------------------------------------

Backend:
- Ejecutado con Uvicorn en un entorno ASGI.
- Configuración de arranque mediante Procfile.
- Versión de runtime definida en runtime.txt para compatibilidad del entorno.
- Endpoints de salud disponibles en /health y /db-check para verificación básica.

Frontend:
- Estructurado como sitio estático con archivos HTML, CSS y JS.
- Diseñado para abrirse directamente desde un navegador o desde un hosting estático.
- Integra la API remota mediante JavaScript y variables de entorno en la lógica del cliente.

Base de datos:
- PostgreSQL como motor principal.
- Modelos relacionales gestionados con SQLAlchemy.
- Migraciones disponibles a través de Alembic.

5.3 VARIABLES Y CONFIGURACIÓN
------------------------------------

La aplicación depende de variables de entorno para su funcionamiento en distintos
eventos. Entre las más importantes se encuentran:
- DATABASE_URL: conexión a la base de datos PostgreSQL
- SECRET_KEY: firma de tokens JWT
- STRIPE_SECRET_KEY: acceso a la API de Stripe
- STRIPE_WEBHOOK_SECRET: verificación de eventos webhook
- FRONTEND_URL: URL base del frontend para redirecciones y CORS
- ANTHROPIC_API_KEY: acceso a la integración de IA
- SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASSWORD y SMTP_FROM: envío de correos

Estas variables deben configurarse de forma segura y no deben exponerse en el código
ni en archivos compartidos públicamente.

5.4 SEGURIDAD E INFRAESTRUCTURA

7. MAPA ESTRUCTURAL DEL PROYECTO PARA IA Y MANTENIMIENTO
================================================================================

Este proyecto está organizado como una aplicación web full-stack con una separación
clara entre frontend, backend, base de datos y documentación. La idea general es que
el frontend entregue una experiencia comercial y visual al usuario, mientras el
backend centraliza la lógica de negocio, seguridad, autenticación, pagos y persistencia.

7.1 ESTRUCTURA GENERAL DEL REPOSITORIO
------------------------------------

- backend/: corazón del sistema. Aquí viven los modelos de datos, rutas API,
  esquemas, servicios, middleware y la entrada principal de la aplicación.
- frontend/: sitio estático que representa la experiencia visible del cliente y del
  administrador. Aquí se encuentra el catálogo, carrito, checkout, páginas
  institucionales y el panel admin.
- docs/: documentación técnica y de negocio. Aquí se concentran notas de proyecto,
  arquitectura y referencias operativas.
- alembic/: migraciones de base de datos para mantener el esquema SQLAlchemy
  alineado con PostgreSQL.
- package.json: configuración mínima de dependencias frontend y soporte de scripts
  de utilidad.

7.2 MAPA DEL BACKEND
------------------------------------

El backend está construido sobre FastAPI y SQLAlchemy, con una organización modular:

- backend/main.py
  Punto de entrada de la aplicación. Aquí se crea la instancia de FastAPI, se
  configuran los middlewares de CORS, se registran los routers principales y se
  exponen endpoints de salud como /health y /db-check.

- backend/database.py
  Define la conexión a PostgreSQL, la sesión de base de datos y el patrón de
  inyección de dependencias que usan las rutas.

- backend/models/
  Contiene los modelos ORM que representan las tablas del negocio. Los modelos más
  importantes son:
  - user.py: usuarios, autenticación y rol de administrador
  - product.py: catálogo de productos y stock
  - category.py: categorías de productos
  - cart.py: carritos y items del carrito
  - order.py: órdenes y detalle de compra
  - layout.py: bloques del editor visual de la tienda
  - product_page.py: contenido enriquecido de páginas de producto
  - settings.py: parámetros de configuración de la tienda

- backend/schemas/
  Define los modelos Pydantic usados para validar y serializar los datos de entrada
  y salida de la API. Cada dominio tiene su propio esquema: user.py, product.py,
  cart.py, order.py, layout.py, category.py, product_page.py.

- backend/routes/
  Agrupa los endpoints HTTP del sistema. Cada archivo representa un dominio
  funcional:
  - auth.py: registro, login y datos del usuario autenticado
  - products.py: CRUD de productos
  - categories.py: categorías y productos por categoría
  - cart.py: agregar, actualizar y eliminar productos del carrito
  - payments.py: checkout con Stripe y webhook de confirmación
  - guest_checkout.py: flujo de compra para usuarios no registrados
  - layout.py: editor visual del layout y generación de bloques con IA
  - product_pages.py: páginas de producto personalizadas
  - metrics.py: métricas de negocio para el panel admin
  - settings.py: ajustes de tienda, colores, banner, slogan y TRM

- backend/services/
  Contiene lógica de negocio transversal que no pertenece directamente a una ruta:
  - auth.py: hash de contraseñas, autenticación y generación de tokens JWT
  - email.py: envío de correos transaccionales y confirmaciones

- backend/middleware/
  - auth.py: middleware y dependencias para verificar JWT y roles de usuario/admin

- backend/alembic/
  Mantiene las migraciones SQL para crear y modificar tablas en PostgreSQL. Es
  fundamental para despliegues y para mantener el estado del esquema alineado.

7.4 FUNCIONES Y FLUJOS IMPORTANTES DEL NEGOCIO
------------------------------------

El proyecto sigue un patrón claro de flujo de negocio que una IA debe entender para
realizar cambios seguros:

1. Autenticación
   - El frontend envía email y contraseña a /auth/login.
   - El backend valida las credenciales, genera un token JWT y lo devuelve.
   - El token se guarda en el cliente y se usa en rutas protegidas.

2. Catálogo y productos
   - /products/ lista productos activos.
   - /products/{id} devuelve detalle de un producto.
   - /categories/ organiza productos por categoría.

3. Carrito
   - /cart/ devuelve el carrito del usuario autenticado.
   - /cart/items permite agregar, actualizar o eliminar productos.
   - La lógica valida stock antes de registrar cambios.

4. Checkout y pagos
   - /payments/checkout crea una orden y una sesión de Stripe.
   - /payments/webhook confirma el pago recibido y actualiza el estado de la orden.
   - Al confirmar, se descuenta stock y se limpia el carrito.

5. Checkout invitado
   - /guest/checkout permite crear un pedido sin crear una cuenta previa.
   - El flujo crea automáticamente un usuario temporal, genera la orden y envía
     correos de bienvenida y confirmación.

6. Layout y edición visual
   - /layout/generate-block genera bloques de contenido con IA.
   - /layout/ y /product_pages/ permiten persistir bloques y contenido personalizado.
   - El frontend admin usa estos endpoints para previsualizar cambios en tiempo real.

7. Ajustes y métricas
   - /settings/ administra branding, colores, banner y configuración comercial.
   - /metrics/dashboard devuelve KPIs para el panel administrativo.

7.5 PUNTOS DE EXTENSIÓN RECOMENDADOS
------------------------------------

Para futuras mejoras o para que una IA trabaje sobre el proyecto sin romperlo, los
puntos más sensibles y recomendables para modificar con cuidado son:

- backend/models/: si se agregan nuevas tablas o columnas, deben reflejarse también
  en Alembic y en los schemas correspondientes.
- backend/routes/: cualquier cambio en endpoints debe ir acompañado de su consumo
  en frontend/js/api.js y en las vistas HTML correspondientes.
- backend/middleware/auth.py: si se cambian permisos o roles, deben revisarse todas
  las rutas protegidas.
- frontend/js/page-blocks.js: el editor visual depende de este archivo para renderizar
  bloques y manejar la experiencia de administración.
- frontend/js/api.js: cualquier cambio en la forma de llamar a la API debe integrarse
  aquí para evitar inconsistencias.

7.6 CONVENCIÓN DE DESARROLLO RELEVANTE
------------------------------------

- El proyecto usa SQLAlchemy en modo síncrono, no async ORM.
- Se prefieren cambios pequeños y explícitos sobre archivos existentes.
- Los secretos y credenciales deben mantenerse en variables de entorno y nunca
  hardcodeadas.
- La separación entre rutas, modelos, esquemas y servicios debe mantenerse para
  facilitar el mantenimiento y la comprensión por parte de herramientas de IA.
