3. ARQUITECTURA TÉCNICA
================================================================================

3.1 BACKEND - FastAPI + PostgreSQL
------------------------------------

BASE DE DATOS (7 TABLAS):

TABLE: users
- id (PK)
- email (UNIQUE)
- hashed_password
- is_admin (default: false)
- created_at
Relaciones: 1-N con carts, 1-N con orders

TABLE: products
- id (PK)
- name
- description
- price (decimal)
- stock (int)
- image_url
Relaciones: N-M con carts (via cart_items), N-M con orders (via order_items)

TABLE: carts
- id (PK)
- user_id (FK, UNIQUE)
- created_at
Relaciones: 1-N con cart_items

TABLE: cart_items
- id (PK)
- cart_id (FK)
- product_id (FK)
- quantity
- subtotal (calculado)
Relaciones: FK cart, FK product

TABLE: orders
- id (PK)
- user_id (FK)
- total (decimal)
- status (PENDIENTE/CONFIRMADA/CANCELADA)
- stripe_session_id
- created_at
Relaciones: 1-N con order_items

TABLE: order_items
- id (PK)
- order_id (FK)
- product_id (FK)
- quantity
- price_at_time (snapshots del precio)
Relaciones: FK order, FK product

TABLE: store_layout
- id (PK)
- block_type (announcement, hero, grid, testimonios, footer)
- content (JSON)
- position (int)
- created_at

TABLE: store_settings
- key (PK)
- value (JSON/string)


MODELOS SQLALCHEMY:

User:
- email único
- password hasheado (bcrypt)
- roles (admin/usuario)

Product:
- nombre, descripción, precio, stock, imagen

Cart:
- carrito único por usuario

CartItem:
- cantidad, subtotal calculado

Order:
- user_id, total, estado, timestamp

OrderItem:
- snapshots de producto al momento del pago

StoreLayout:
- bloques con tipo, contenido, posición

StoreSetting:
- key-value pairs


RUTAS API (5 ROUTERS):

ROUTER: /auth
- POST /register        → Crear usuario (crea carrito automáticamente)
- POST /login          → Retorna JWT token
- GET /me              → Datos usuario actual (JWT)

ROUTER: /products
- GET /                → Listar todos los productos (público)
- POST /               → Crear producto (admin)
- PUT /{id}            → Actualizar producto (admin)
- DELETE /{id}         → Eliminar producto (admin)

ROUTER: /cart
- GET /                → Obtener carrito actual + items + total (JWT)
- POST /items          → Agregar item al carrito (JWT, valida stock)
- PUT /items/{id}      → Ajustar cantidad (JWT)
- DELETE /items/{id}   → Remover item (JWT)

ROUTER: /payments
- POST /checkout       → Crear sesión Stripe, crea orden PENDIENTE (JWT)
- POST /webhook        → Webhook Stripe (sin auth, verifica firma)

ROUTER: /layout
- GET /                → Obtener bloques del layout (público)
- POST /               → Crear bloque (admin)
- PUT /{id}            → Actualizar bloque (admin)
- DELETE /{id}         → Eliminar bloque (admin)


SERVICIOS PRINCIPALES:

auth.py:
- Hash bcrypt para passwords
- JWT HS256 (30 minutos de expiración)
- Validación de credenciales
- Generación de tokens

Cart Logic:
- Validación de stock disponible
- Cálculo automático de totales
- Manejo de cantidades
- Eliminación de items

Payments:
- Creación de checkout sessions en Stripe
- Webhook verification (verifica firma HMAC)
- Actualización de estado de órdenes
- Manejo de cancelaciones

Layout Service:
- CRUD de bloques dinámicos
- Renderizado de defaults
- Integración con IA Anthropic

IA Integration:
- Llamadas a Anthropic Claude
- Generación de HTML/CSS desde prompts
- Solo accesible por admins


AUTENTICACIÓN Y AUTORIZACIÓN:

- JWT Bearer Token en header Authorization
- Token de corta duración (30 min)
- Middleware get_current_user() protege endpoints de usuario
- Middleware get_current_admin() protege endpoints admin
- Flag is_admin en tabla users para control de acceso
- Al registrarse, se genera carrito vacío automáticamente

4. FLUJOS PRINCIPALES
================================================================================

4.1 FLUJO DE COMPRA (CUSTOMER):

1. REGISTRO
   → Usuario completa email y password
   → Backend valida, hashea password, crea usuario
   → Carrito vacío se genera automáticamente
   → Se retorna JWT token
   → Frontend guarda token en localStorage
   → Redirecciona a index

2. EXPLORACIÓN
   → Usuario entra a index.html
   → JavaScript hace GET /products
   → Obtiene lista completa de productos
   → Renderiza grid de cards

3. AGREGAR CARRITO
   → Click en botón "Agregar al carrito"
   → POST /cart/items {product_id, quantity}
   → Backend valida:
      - Usuario autenticado (JWT válido)
      - Producto existe
      - Stock disponible
   → Crea CartItem, calcula subtotal
   → Retorna carrito actualizado
   → Frontend muestra confirmación

4. VER CARRITO
   → Usuario navega a cart.html
   → GET /cart (con JWT)
   → Backend retorna:
      - Array de cart_items con detalles del producto
      - Total calculado
   → Frontend renderiza tabla con items

5. CHECKOUT
   → Click en "Proceder a pagar"
   → POST /payments/checkout (con JWT)
   → Backend:
      - Crea orden con status PENDIENTE
      - Llama Stripe.checkout.Session.create()
      - Retorna session.url
   → Frontend redirige a session.url (Stripe)

6. PAGAR EN STRIPE
   → Stripe muestra formulario de pago
   → Usuario completa tarjeta
   → Stripe procesa pago

7. CALLBACK STRIPE
   → Stripe redirige a checkout.html?success=true
   → Frontend muestra página de éxito

8. WEBHOOK STRIPE (ASÍNCRONO)
   → Stripe envía evento payment_intent.succeeded
   → POST /payments/webhook
   → Backend:
      - Verifica firma HMAC
      - Busca orden por stripe_session_id
      - Actualiza status a CONFIRMADA
      - Vacía carrito
   → Éxito: Orden guardada en BD
   → Carrito listo para próxima compra


4.2 FLUJO ADMIN (MANAGEMENT):

1. LOGIN COMO ADMIN
   → Email y password con is_admin=true
   → JWT retornado
   → Token guardado en localStorage

2. ACCESO ADMIN.HTML
   → Usuario navega a admin.html
   → Frontend verifica get_current_admin() en GET /layout
   → Si no es admin, redirige a index

3. CRUD PRODUCTOS
   → POST /products: crear nuevo producto
   → PUT /products/{id}: actualizar
   → DELETE /products/{id}: eliminar
   → GET /products: listar todos

4. EDITOR LAYOUT
   → GET /layout: obtiene bloques actuales
   → Frontend renderiza lista de bloques
   → Opciones:
      a) AGREGAR BLOQUE: elije tipo (announcement, hero, grid, etc)
      b) EDITAR BLOQUE: modifica propiedades
      c) GENERAR CON IA: ingresa prompt
         → POST a endpoint IA
         → Claude genera HTML/CSS
         → Renderiza preview

5. GUARDAR CAMBIOS
   → PUT /layout/{id} o POST /layout
   → Backend actualiza BD
   → Cambios reflejados en index.html

6. PREVIEW EN TIEMPO REAL
   → Mientras edita, ve cambios en preview
   → HTML/CSS renderizado instantáneamente


4.3 FLUJO DE CARRITO:

CREAR:
POST /cart/items
{
  "product_id": 1,
  "quantity": 2
}
→ Backend crea CartItem
→ Valida stock: product.stock >= quantity
→ Calcula subtotal: product.price * quantity
→ Retorna carrito actualizado

LEER:
GET /cart
→ Backend retorna objeto:
{
  "id": 1,
  "items": [
    {
      "id": 1,
      "product_id": 1,
      "quantity": 2,
      "subtotal": 50.00,
      "product": {nombre, precio, imagen}
    }
  ],
  "total": 50.00
}

ACTUALIZAR:
PUT /cart/items/{id}
{"quantity": 3}
→ Backend valida nuevo stock
→ Recalcula subtotal y total
→ Retorna carrito actualizado

ELIMINAR:
DELETE /cart/items/{id}
→ Backend elimina CartItem
→ Recalcula total
→ Retorna carrito actualizado

VALIDACIÓN:
Cada operación verifica:
- Usuario autenticado (JWT válido)
- Producto existe
- Stock disponible >= cantidad solicitada
- CartItem pertenece al carrito del usuario


5. INTEGRACIONES EXTERNAS
================================================================================

5.1 STRIPE 

API Integration:
- Modelo: Stripe Python library
- Operación principal: stripe.checkout.Session.create()

Flujo de Pagos:
1. Backend crea orden con status PENDIENTE
2. Llama stripe.checkout.Session.create() con:
   - line_items: productos del carrito
   - metadata: order_id
   - success_url, cancel_url
3. Retorna session.url
4. Frontend redirige a URL de Stripe
5. Usuario completa pago en Stripe
6. Stripe redirige de vuelta (success_url o cancel_url)

Webhooks:
- Stripe envía eventos a POST /payments/webhook
- Backend verifica firma HMAC (secret_key)
- Evento escuchado: payment_intent.succeeded
- Acción: actualizar orden a CONFIRMADA, vaciar carrito

Estado: ✅ 100% funcional, webhooks verificados


5.2 ANTHROPIC CLAUDE 

Configuración:
- Modelo: claude-sonnet-4-6

Uso:

Restricciones:
